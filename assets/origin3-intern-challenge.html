What is this?

Welcome to your first real engineering challenge. This isn't a school project where you submit code and get a grade. This is closer to how software actually gets built — with tickets, branches, reviews, and teammates who depend on you.

You're building LibTrack, a library book borrowing system. Simple problem. The hard part isn't the code — it's doing everything the right way, every day, for a month.

There are two teams. Same problem. At the end, you both present to the founders. The team with the better process wins — not necessarily the prettier UI.

The real lesson

Most fresh grads can write code. Very few can break a problem into tasks, track their work properly, review each other's code, and ship something that actually works. That's what this month is about.

What about the capstone?

At the end of the program, each team produces a Technical Proposal — a document written as if you're pitching LibTrack to a real client. It covers what you built, the technical decisions you made, why you made them, how the system is architected, and what you'd do differently if you built it again.

This is a document deliverable, not more code. It's due after Demo Day and is separate from the competition scoring. Think of it as your exit artifact — proof that you can communicate engineering decisions in writing, not just in code. More details will be shared in Week 4.

The problem

LibTrack — Library Book Borrowing System

Your school library is still tracking book borrowing on paper. The librarian is drowning in index cards. Students have no idea which books are available without walking to the library.

Your job is to build a simple web system that fixes this.

A student can: Browse available books · Submit a borrow request · See their request was received

A librarian can: See all borrow requests · Mark a book as Borrowed · Mark it as Returned (so it becomes available again)

That's it. No login system required. No fancy features. Keep the scope tight and ship it properly.

Jira setup + guide

Jira is where your work lives. Each team sets up their own Jira project from scratch — it's part of the Week 1 planning work. Before you touch any code, every task needs to exist as a ticket.

Step-by-step: setting up your Jira project

Each team creates their own completely separate Jira account and project. The team lead does this on Day 1 of Week 1, then invites the rest of the team. Follow every step in order.

01
Create a free Atlassian account — go to atlassian.com/software/jira and click Get it free. Use your personal email. One account per team lead — the rest of the team gets invited in Step 4.
02
Create a new Jira Software project — after signing in, click Create project. Choose Scrum as the project type. Name it LibTrack - Team A or LibTrack - Team B. Set the project key to LTA for Team A or LTB for Team B. This prefixes all your ticket IDs — so Team A has LTA-01, LTA-02... and Team B has LTB-01, LTB-02. Keeping them distinct makes cross-team discussion in Discord unambiguous.
03
Configure your board columns — go to Project Settings → Board. You need exactly four columns: Backlog, In Progress, For QA, Done. Delete any default columns that don't match. Drag to reorder if needed.
04
Invite your teammates — go to Project Settings → Access. Invite each teammate by their email address. Set their role to Member. Everyone must accept the invite before they can create or update tickets.
05
Create your first sprint — in the Backlog view, click Create Sprint. Name it Sprint 1 and set the duration based on your sprint plan. Leave it in draft — do not click Start Sprint. You fill the backlog first (Step 6), then start the sprint only after founders approve your Week 1 plan.
06
Write your tickets — create all your tickets in the Backlog (not inside the sprint yet). Use the format from the example below. Once all tickets are written, drag the ones for Sprint 1 into the sprint container. Do not click Start Sprint yet — that happens after founders approve your plan at the end of Week 1.
07
Share the board link with founders — copy your project URL and drop it in Discord. Founders need access to your board to review it as part of the Week 1 gate. Make sure the project is set to Private but with founders added as viewers.
Common mistake

Don't start your sprint before all your tickets are written and reviewed. Once a sprint starts, Jira locks the structure. Write everything first, then start the sprint at the end of Week 1 when your plan is approved.

You'll need tickets for: backend endpoints, frontend pages/components, connecting frontend to backend, QA test cases, and demo preparation. The exact breakdown is yours to decide.

Your daily Git workflow

A Pull Request (PR) is how you ask your teammates to review your code before it merges into main. Think of it as saying "I'm done with this — can someone check it before we ship?" It's not optional. It's how the team catches mistakes before they become everyone's problem.

# 1. Always start from the latest main
git checkout main
git pull origin main

# 2. Create your branch for the ticket you're working on
git checkout -b feature/LTA-02-get-books-endpoint

# 3. Do your work. Commit often with clear messages.
git add .
git commit -m "LTA-02: add GET /books endpoint returning all books"
git commit -m "LTA-02: add availability status field to book response"

# 4. Push your branch and open a Pull Request on GitHub
git push origin feature/LTA-02-get-books-endpoint

# 5. On GitHub → New Pull Request → fill in the PR template
# 6. Tag a teammate to review it — don't merge your own PRs

Tech stack — your choice

You decide what to build with. There are no wrong answers — but there are unjustified answers. Every choice needs a reason. "Everyone uses it" is not a reason. "It fits our data model because..." is a reason.

Questions to answer per layer

Frontend

Plain HTML/CSS/JS? A framework like React, Vue, or Angular? How complex is the UI? Does the complexity justify a framework, or would vanilla JS be faster to ship?

Backend

What language? What framework? How familiar is your team with it? What does it do well for an API like this? How easy is it to set up locally on everyone's machine?

Database

SQL or NoSQL? Which one? Why does LibTrack's data fit that model? See the Database choice section for a full breakdown.

Deployment

Where does this run? Free tier services? Your own machine for demo? How does the frontend reach the backend in production? What environment variables do you need?

Database choice

This is one of the biggest decisions you'll make during Week 1. Think it through carefully and write an ADR for it.

Your options

PostgreSQL
Relational · SQL
Structured data with clear relationships
Transactions — prevent double-booking a book
Strong data integrity
Needs more setup than NoSQL
Schema must be defined upfront
MySQL / SQLite
Relational · SQL
SQLite needs zero setup — file-based
MySQL widely used, lots of tutorials
Same SQL skills as PostgreSQL
SQLite not for production use
MySQL slightly less featured than Postgres
MongoDB
Document · NoSQL
Flexible schema — easy to change
Fast to start, JSON-like documents
LibTrack data is relational — does it fit?
No native transactions in older versions
You'll need to justify why NoSQL fits here

Questions to guide your decision

Think about this What does LibTrack's data actually look like? You have Books, Students, and Borrow Requests. How do these relate to each other? If a book can only be borrowed once at a time, how does your database enforce that? What happens if two students submit a borrow request at the exact same millisecond?
And this Is your data structured and predictable, or flexible and varied? Do you need to JOIN tables together to answer common questions? Or is each piece of data self-contained? Your answers should point you toward SQL or NoSQL.
No wrong answer — but you must defend it

If you pick MongoDB, explain why document storage fits LibTrack's data model. If you pick PostgreSQL, explain why relational structure and transactions matter here. Founders will ask. "Because I know it" is a partial answer — pair it with a technical reason.

Writing an ADR

An ADR (Architecture Decision Record) is a short document that captures a technical decision — what you chose, why, and what you considered. Real engineering teams write these so future developers understand why things are built the way they are.

You write one ADR per major decision. At minimum: database choice and backend framework. You can write more if you made other significant choices.

Example ADR — database choice

ADR-001 · Database selection · Team A · [Date]

Status
Accepted
Decision
We will use PostgreSQL as our database.
Context
LibTrack has three clearly related entities: Books, Borrow Requests, and Students. A book's availability status depends on whether an active borrow request exists for it. We also need to prevent two students from borrowing the same book simultaneously.
Reasoning
Our data has clear relationships and we need transactional integrity. PostgreSQL's ACID transactions ensure that if two requests arrive simultaneously, only one can succeed. The relational model maps naturally to our entities and their relationships.
Alternatives considered
MongoDB — rejected because our data is relational, not document-shaped. There's no natural "document" here that stands alone. Forcing it into MongoDB would require manual relationship management that SQL handles natively.
Consequences
We need to define our schema upfront and run migrations if it changes. We'll need to set up a local Postgres instance or use a free hosted service like Supabase or Railway.

Your ADR skeleton — fill this in

ADR-00? · [What decision] · [Your team] · [Date]

Status
Accepted / Proposed / Superseded
Decision
We will use ___ because ___
Context
What problem are you solving? What constraints exist? What does LibTrack actually need from this layer?
Reasoning
Why does your choice fit the context? Be technical. What specific feature or property made you choose this?

Testing & QA

Every feature needs test cases before it's considered done. QA isn't something you do at the end — it's woven into the whole sprint. The QA intern on your team writes and executes test cases. Everyone is responsible for making sure their own work can be tested.

What is cross-testing?

Cross-testing means you test a feature you didn't build. If you wrote the borrow form, you don't test it — a teammate does. This matters because the person who built it already knows how it's supposed to work. Someone fresh will find the edge cases they never thought of.

QA tips

Think beyond the happy path. What happens when a student submits with an empty form? What if someone tries to borrow an already-borrowed book? What if the backend is down when the frontend loads? Edge cases are where bugs live.

How you're scored

The team with the better UI doesn't automatically win. The team with the better process wins. Here's exactly how Demo Day is judged.

40%

25%

Test cases written · Edge cases covered · Bugs found and fixed with tickets
25%
Working demo
All 7 acceptance criteria met · Frontend talks to real backend · No hardcoded data
10%
Demo clarity

Read this carefully

40% of your score is pure process. You can have a beautiful app and still lose if your Jira board is a mess, your commits have no messages, and your PRs have no descriptions. The code is only half the job.

Non-negotiables

These rules apply every day, both teams, no exceptions. They're not suggestions.

  • No direct commits to main — ever. Every change lives on a branch. Every branch ends in a PR. Every PR gets reviewed by a teammate before it merges.
  • No Jira ticket = the work didn't happen. If you built something that isn't tracked in Jira, it doesn't count. Create the ticket first, then code.
  • No ticket moves to Done without QA sign-off. You don't decide when your own ticket is done. The QA intern does.
  • No coding during Week 1. The entire first week is planning. If you're writing code before Week 2, you're doing it wrong.
  • Daily Discord standup every morning. Short. Three lines. Yesterday / Today / Blockers. If you miss it, message the team why.
  • Ask for help early. If you're stuck for more than 30 minutes, say something in Discord. Staying stuck silently hurts the whole team.
  • You own your decisions. If you chose MongoDB and it causes problems later, you solve it. The founders won't bail you out of a technical choice — but they will help you think it through if you ask early enough.

One last thing

This challenge is designed to be hard enough to learn from and simple enough to actually finish. The scope is tight on purpose. Don't add features. Don't gold-plate the UI. Ship what's in the acceptance criteria, do it with discipline, and you'll be ahead of most fresh graduates who've never had to work this way before.

Good luck. May the better process win.
